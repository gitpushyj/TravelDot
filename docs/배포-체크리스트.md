### 1. `public.users.tier` 클라이언트 변경 차단 — ✅ 해결됨 (2026-05-12)

**적용 내용**

`supabase/migrations/20260512100000_lock_tier_and_home_country.sql`로 처리.
- `revoke update (tier) on public.users from authenticated, anon;` 적용 완료
- 이후 `tier`는 결제 webhook(service_role) 및 Supabase 대시보드에서만 변경 가능
- Table Editor / SQL Editor에서 dev 토글은 그대로 가능 (service_role)

**검증 결과**

`information_schema.column_privileges`에서 `tier` 컬럼의 authenticated/anon UPDATE 권한이 사라짐을 확인.

---

<details>
<summary>원본 메모 (참고용)</summary>

**현재 상태 (위험)**

`public.users`의 RLS update 정책이 `id = auth.uid()`만 검사한다. 즉 인증된 사용자가 자기 행의 `tier`를 클라이언트에서 임의로 바꿀 수 있다.

```ts
// 클라이언트에서 이게 그냥 통한다
await supabase.from("users").update({ tier: 2 }).eq("id", myId);
```

**왜 지금 두는가**

`tier`를 직접 토글해 무료/premium/power 동작을 빠르게 테스트하기 위해 dev 단계에선 의도적으로 열어둠.

**놓치면**

누구나 클라이언트에서 자기 tier를 power로 올리고 모든 유료 기능을 무료로 쓸 수 있음. **매출 손실 + 정책 우회**.

**결제 붙일 때 반드시 적용 (택일)**

**A) column-level revoke (추천)**
```sql
revoke update (tier) on public.users from authenticated;
revoke update (tier) on public.users from anon;
-- 이후 service_role(결제 webhook)만 tier 변경 가능
```

**B) trigger로 차단**
```sql
create or replace function public.users_block_tier_change()
returns trigger language plpgsql as $$
begin
  if old.tier is distinct from new.tier
     and current_setting('request.jwt.claim.role', true) <> 'service_role' then
    raise exception 'tier change not allowed for non-service role';
  end if;
  return new;
end;
$$;

create trigger users_block_tier_change
  before update on public.users
  for each row execute function public.users_block_tier_change();
```

**검증 방법**

```ts
// 일반 로그인 세션으로 아래가 throw하면 OK
await supabase.from("users").update({ tier: 2 }).eq("id", myId);
```

</details>

### 2. `public.users.home_country_changed` 클라이언트 변경 차단 — ✅ 해결됨 (2026-05-12)

**적용 내용**

`supabase/migrations/20260512100000_lock_tier_and_home_country.sql`에 같이 묶어 처리.
- 단순 revoke 대신 `users_home_changed_one_way` BEFORE UPDATE 트리거로 `true → false` 되돌림만 차단
- 사유: 클라이언트가 `markHomeCountryChangedInDb`에서 `false → true` 정상 흐름을 그대로 수행해야 하므로 컬럼 권한 자체는 유지

**검증 결과**

트리거 `users_home_changed_one_way`가 BEFORE UPDATE로 등록됨을 확인.

---

<details>
<summary>원본 메모 (참고용)</summary>

**현재 상태 (위험)**

`tier`와 같은 사유로 사용자가 자기 `home_country_changed`를 `false`로 되돌리면 "본국 1회 변경" 제약을 무한 우회 가능.

**해결**

위 #1과 동일한 패턴(column-level revoke 또는 trigger). `tier` 막을 때 같이 처리.

```sql
revoke update (home_country_changed) on public.users from authenticated;
revoke update (home_country_changed) on public.users from anon;
```

이렇게 하면 `markHomeCountryChangedInDb`도 service_role을 거치도록 RPC로 옮기거나, `home_country_changed`를 별도 RPC 함수로 갱신하게 바꿔야 함.

**대안 (간편)**: `home_country_changed`를 `false → true`로만 가는 쪽으로만 trigger에서 허용.

```sql
create or replace function public.users_home_changed_one_way()
returns trigger language plpgsql as $$
begin
  if old.home_country_changed = true and new.home_country_changed = false then
    raise exception 'home_country_changed cannot revert to false';
  end if;
  return new;
end;
$$;

create trigger users_home_changed_one_way
  before update on public.users
  for each row execute function public.users_home_changed_one_way();
```

</details>

## 🟡 Should — 출시 전 권장

### 3. GraphQL introspection에서 `trips`/`users` 가리기

**현재 상태**

Supabase advisor가 `trips`, `users` 테이블이 `anon`/`authenticated` 역할에 GraphQL 스키마로 노출된다고 경고. 데이터 자체는 RLS가 막아서 **유출 위험은 없음** — 다만 테이블 존재·컬럼명이 introspection으로 보임.

**해결**

```sql
revoke select on public.users, public.trips from anon;
-- authenticated는 RLS 정책이 자기 행만 보여주므로 select 권한은 유지
```

`anon` 권한만 revoke하면 비로그인 사용자에게 스키마가 안 보임. 같이 점검해야 할 함수:
- `public.handle_new_user`는 SECURITY DEFINER인데 `anon`/`authenticated`가 RPC로 호출 가능 (advisor 경고). 트리거 전용이니 `revoke execute on function public.handle_new_user from anon, authenticated` 같이 처리.

### 5. 동기화 게이팅 (`tier`) 클라이언트 검증 보강

**컨텍스트**

`tier < 1`(무료)은 pull 안 한다는 클라이언트 게이팅이 들어갈 예정. 클라이언트가 거짓말로 `tier=1`이라고 우기고 pull을 시도해도 RLS가 자기 행만 읽게 막으니 **데이터 유출 위험은 없음**. 다만 무료 사용자가 클라이언트 코드 패치로 pull을 강행해 데이터를 받아 쓸 수는 있음 (자기 데이터지만 정책 우회).

**대응**

크리티컬은 아님. 보강하려면 RPC 함수에 `tier >= 1` 검사를 두고 그 RPC만 노출.

### 6. 결제 webhook의 service_role 키 안전 보관

`tier` 갱신을 service_role로 해야 하는 만큼, webhook 환경 변수는 절대 클라이언트 번들에 들어가면 안 됨.
- Supabase Edge Function 또는 별도 백엔드에서만 사용
- `EXPO_PUBLIC_*` prefix 절대 금지
