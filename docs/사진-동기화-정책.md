# 사진 동기화 정책

> 사진을 "어떻게 기록하고, 무엇을 동기화할지"에 대한 설계 결정 문서

---

## 결론

- **사진 원본은 서버에 올리지 않는다.** 항상 디바이스 PhotoLibrary에 그대로 둔다.
- **추후 서버 동기화가 도입될 경우, 동기화 대상 사진은 방문(날짜)당 최대 3장으로 한정한다.**
- **그 외 사진은 디바이스 PhotoLibrary에서 날짜로 매칭하여 "전체사진보기"로 노출한다.**

---

## 배경: 왜 서버에 사진을 올리지 않기로 했나

1. **불필요함**: 본 앱의 사진은 사용자의 PhotoLibrary에 이미 존재한다. 서버에 다시 저장할 동기는 약하다.
2. **비용/프라이버시**: 사진 1장당 수 MB. 사용자가 같은 날짜에 100장 이상 찍는 경우도 흔하다. 모두 업로드하면 스토리지 비용이 빠르게 커지고, 사용자도 자기 사진이 어디까지 올라갔는지 추적하기 어렵다.
3. **현재 3장 제한의 진짜 이유**: "혹시 서버를 붙이게 됐을 때 너무 많은 사진이 올라가는 게 부담스러워서"였다. 즉 제한은 *서버를 가정한* 제약이지 *기능적 가치*가 아니다.

→ 따라서 **로컬 기록은 무제한**으로 풀고, **"서버에 갈 수도 있는 후보"만 3장으로 묶는다**.

---

## 식별자 문제 (왜 사진 ID로 동기화하지 않는가)

플랫폼 사진 식별자는 동기화 키로 부적합하다:

| 식별자 | 깨지는 시점 |
| --- | --- |
| iOS `PHAsset.localIdentifier` | iOS → Android 기기 변경, 재설치, 백업 복원 |
| Android `MediaStore` ID | Android → iOS 기기 변경, 재설치, SD 카드 변경 |
| EXIF (촬영 시각 + GPS) "지문" | EXIF 누락/편집 사진, 새 기기에 사진 자체가 옮겨지지 않은 경우 |

따라서 **사진 식별자만 동기화하는 방식은 신뢰할 수 없다.** 핵심 데이터(방문 기록 자체)는 사진 식별자에 의존하지 않게 분리한다.

---

## 데이터 계층화

| 계층 | 내용 | 양 | 동기화 |
| --- | --- | --- | --- |
| **방문 기록** | 국가코드, 날짜, 메모 | 무제한 | ✅ 항상 (텍스트 메타데이터만) |
| **대표 사진** | 방문 1건당 최대 3장의 이미지 원본 | 방문당 ≤ 3 | ✅ 옵트인 동기화 (이미지 자체 업로드) |
| **그 외 사진** | 해당 날짜의 PhotoLibrary 사진 전부 | 무제한 | ❌ 동기화 안 함 (디바이스에서 직접 조회) |

핵심: **"진짜 가치 있는 데이터(방문 기록)"는 가볍고 안전하게, "이미지 원본"은 최소한만 운반한다.**

---

## 대표 사진 3장 선정 규칙

- **자동 스캔 시점**: 해당 날짜의 PhotoLibrary 사진 중 **촬영 시각 기준 가장 최근 3장**을 자동으로 대표 사진 후보로 등록.
- **사용자 선택**: 날짜 상세 화면에서 사용자가 직접 3장을 다시 고를 수 있다.
- **3장을 초과해 선택 시도**: 기존 선택 1장을 빼야 추가 가능 (UI에서 강제).

> 3이라는 숫자는 "썸네일 그리드 한 줄에 들어가는 양 + 서버 저장 부담의 상한"으로서의 의미다. 추후 운영하며 조정 가능.

---

## 날짜 상세 페이지 UX

```
[2026-04-15 · 일본]
┌─────────────────────────────┐
│  [대표사진 1] [2] [3]        │  ← 앱 데이터 (최대 3장)
│                              │
│  메모: ...                   │
│                              │
│  [ 전체사진보기 ]            │  ← 클릭 시 PhotoLibrary 조회
└─────────────────────────────┘
```

- **대표 사진 영역**: 3장. 항상 즉시 표시 (앱 내 데이터).
- **"전체사진보기" 버튼**: 클릭 시 해당 날짜에 촬영된 PhotoLibrary 사진 전체를 조회해 보여준다.
  - 권한이 없거나, 기기 변경으로 PhotoLibrary가 비어 있으면 "표시할 사진이 없습니다" 안내.
  - 동기화 대상이 아니므로 기기 간 결과가 다를 수 있음 (의도된 동작).

---

## 향후 서버 동기화 시나리오

1. 사용자가 동기화 옵트인.
2. 앱은 **방문 기록(메타데이터) + 대표 사진 3장**을 서버에 업로드.
3. 새 기기 설치 시:
   - 방문 기록 + 대표 사진 3장 → 즉시 복원.
   - "전체사진보기" → 새 기기의 PhotoLibrary에 해당 날짜 사진이 있을 경우에만 표시 (없어도 방문 기록 자체는 손실되지 않음).

이 모델에서는 **기기 변경/재설치 시에도 핵심 데이터(방문 기록 + 대표 3장)는 보존**되며, "전체사진"은 best-effort로 취급한다.

---

## 미해결 / 추후 결정

- 대표 사진 저장 형식: 원본 vs 압축본(예: 2048px JPEG). 서버 도입 시 비용을 고려해 결정.
- "전체사진보기"의 정렬·필터: 시각순? 위치 일치 여부 표시?
- 사용자가 대표 3장을 비워둘 수 있게 할지(0장 허용) 여부.
